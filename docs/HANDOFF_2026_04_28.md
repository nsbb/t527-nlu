## 3. 아키텍처 요약

### 3.1 파이프라인

```
STT 발화 → preprocess → ONNX 추론 → post_rules → DST → response_v2 → 한국어 응답
```

### 3.2 모델 구조 (CNNMultiHead)

```
입력: token_ids (max_len=32)
  → ko-sbert WordPiece embedding (768dim, freeze)
  → Linear(768→256) + Dropout
  → CNN 4L (k=3,5,7,3, residual, BN, ReLU)
  → Global Mean Pool → [B, 256]
  → 5 heads: fn(20), exec(5), dir(9), param_type(5), judge(5)
```

**주의**: param_type, judge head는 모델이 예측하지만 파이프라인에서 완전히 무시됨. 모든 케이스 rule로 강제 override.

### 3.3 앙상블 방식

두 ONNX(v28, v72)를 단일 그래프로 통합. head별로 다른 모델 출력 선택.

- flat intent(136+) vs **multi-head(fn×exec×dir)** 비교 → multi-head 채택

## 7. 핵심 알아야 할 사항 (함정 방지)

### 7.1 데이터 관련

- **소규모 패치 데이터 추가 = 항상 regression**: v29~v33, v58에서 반복 확인. 전체 데이터 재학습만 안전.
- **v70 대규모 라벨 수정 실패**: 1,180건 수정이 오히려 성능 -4%p. 부분적 수정은 데이터 일관성 파괴. 수정 전 꼭 검증.
- **param_type / judge head**: 학습은 되지만 파이프라인에서 완전 무시. 수정해도 의미 없음.

### 7.2 앙상블 관련

- **v28과 v46/v71/v72 weight space 비볼록**: Model Soup(weight 평균) 실패. prediction-level 앙상블만 가능.
- **head별 최적 모델이 다름**: fn은 최신 모델, exec/dir은 v28. 일괄 교체 금지.

### 7.3 NPU 관련

- **T527 NPU에 NLU 올리기 불가**: 9가지 방식 전부 시도, 전부 정확도 0. CPU ONNX가 확정 배포 방식.
- STT(Conformer) NB 배포는 성공 — 그건 NeMo Docker 파이프라인이라 NLU와 완전 다른 문제.

### 7.4 모델 선택

- **TS(Test Suite) vs KE(KoELECTRA) 트레이드오프**: 구조적 한계. TS에서 96%이면 KE 75%, KE에서 97%면 TS 91%. 앙상블로 94%/97% 균형 달성.
- **CNN > Conformer at 24.5K**: 데이터 적으면 간단한 구조가 낫다. Transformer 계열 실험 비권장.

### 7.5 응답 생성

- **SPECIFIC_PATTERNS 순서 중요**: regex 매칭은 순서대로, 먼저 나오는 것이 우선. 새 패턴 추가 시 위치 주의.
- **"사우나" 단독 키워드 매칭**: 이전 `r'사우나\s*(?:같|야|네)'` 패턴이 "사우나가 따로 없어" 미매칭. 현재는 `r'사우나'`로 수정됨.

```python
    l28['exec_type'],    # exec: v28 (안정적)
    l72['param_direction'],  # dir: v72 (라벨 수정)
    l28['param_type'],   # param: v28
    l72['judge'],        # judge: v72
)
```

head별로 어느 모델 출력을 쓸지 조합 변경 가능.
