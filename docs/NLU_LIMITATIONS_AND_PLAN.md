# 분석: NLU의 구조적 한계와 해결 방법

## 질문

"찜통이야"→ac_on은 학습 데이터에 넣어서 되는 거다.
그러면 학습에 없는 새로운 간접 표현("이 방이 사우나 같아")은 안 되는 거 아닌가?
매번 재학습은 말이 안 되는데, LLM 말고 해결 방법이 있는가?

## 답: 맞다. 구조적 한계는 있다. 하지만 생각보다 심각하지 않다.

### 1. 현실적으로 간접 표현은 유한하다

스마트홈에서 "덥다"를 표현하는 방법:
```
학습에 있는 것: 찜통이야, 더워, 더워 죽겠어, 땀이 나, 후덥지근해, 열대야, 무더워
학습에 없는 것: 사우나 같아, 불가마야, 한증막이야, 뜨거워, 푹푹 찐다
```

"덥다"의 한국어 표현은 무한하지 않다. 20~30개면 거의 다 커버된다.
51개 intent × 평균 5개 간접 표현 = ~250개만 만들면 실사용의 95%+ 커버.

**이건 LLM도 마찬가지다.** LLM이 "찜통이야"를 이해하는 건 학습 데이터에 비슷한 패턴이 있었기 때문이지, "이해"하는 게 아니다. 다만 LLM은 학습 데이터가 수조 개라서 커버리지가 넓을 뿐.

### 2. 해결 방법 3가지 (LLM 없이)

#### 방법 A: LLM으로 학습 데이터를 만든다 (추론이 아니라 데이터 생성)

```
[오프라인, 1회성]
GPT/Claude에게: "에어컨 켜달라는 의미의 간접 표현 100개 만들어줘"
→ "사우나 같아", "불가마야", "한증막이야", "푹푹 찌네", ...

이걸 학습 데이터에 넣고 CNN 재학습 (30초)
```

**LLM은 추론에 쓰는 게 아니라 데이터 생성에 쓴다.**
한 번만 하면 되고, 이후 추론은 CNN(1ms)으로.

#### 방법 B: Sentence Embedding 유사도 (fallback)

```
CNN 추론 → confidence 낮으면 → ko-sbert-sts로 유사도 비교

"사우나 같아" → CNN: confidence 30% (모르겠음)
             → ko-sbert-sts: "찜통이야"와 유사도 0.85 → ac_on
```

이미 테스트함: ko-sbert-sts CPU 12ms, 간접 표현 95.7%.
CNN(1ms) + fallback(12ms) = 최악 13ms. 충분히 빠름.

#### 방법 C: 운영 중 로그 수집 → 주기적 재학습

```
1주일 운영 → 실패 로그 수집 → "사우나 같아"(미인식) 발견
→ 학습 데이터에 추가 → 재학습 (30초) → OTA 업데이트
```

이건 모든 ML 시스템의 표준 운영 방식이다. "매번 재학습"이 아니라 "주기적 개선".

### 3. LLM이 필요한 진짜 경우

| 상황 | NLU | LLM |
|------|-----|-----|
| "찜통이야" (학습에 있는 표현) | **O (1ms)** | O (1~3초) |
| "사우나 같아" (학습에 없지만 유사한 표현) | fallback으로 O (13ms) | O (1~3초) |
| "아까 말한 거 취소해줘" (대화 맥락 필요) | **X** | **O** |
| "에어컨이랑 공기청정기 차이가 뭐야" (자유 질문) | **X** | **O** |
| "오늘 저녁 뭐 해먹지" (스마트홈 밖 질문) | **X** | **O** |

**NLU의 진짜 한계는 "안 본 표현"이 아니라 "대화 맥락"과 "자유 질문"이다.**
간접 표현은 데이터로 해결 가능. 맥락 추론은 NLU로 불가능.

### 4. 권장 아키텍처

```
사용자 발화
    │
    ▼
[CNN NLU] (NPU, 1ms)
    │
    ├─ confidence ≥ 0.7 → intent 확정 → 기기 제어
    │
    └─ confidence < 0.7 → [ko-sbert-sts fallback] (CPU, 12ms)
                              │
                              ├─ 유사도 ≥ 0.6 → intent 확정
                              │
                              └─ 유사도 < 0.6 → "잘 이해하지 못했어요"
                                                (로그 저장 → 주기적 재학습)
```

## 결론

- **"매번 재학습"은 필요 없다.** LLM으로 한 번에 대량 생성하면 95%+ 커버.
- **나머지 5%는 fallback(sentence embedding)으로 처리.**
- **진짜 LLM이 필요한 건 "대화 맥락"이지 "간접 표현"이 아니다.**
- 구현할 거면: CNN + confidence threshold + ko-sbert-sts fallback 파이프라인.

# 추가 이슈: 오버피팅 + 데이터 부족

## 문제 1: 테스트 데이터 84.6%가 학습 데이터와 겹침

현재 test_v10.csv 682개 중 577개가 학습 데이터에 있다.
100% 정확도는 의미 없다. **학습에 없는 발화로 테스트해야 진짜 성능.**

## 문제 2: 학습 데이터 3,009개는 94개 intent 기준 너무 적다

- 94개 intent ÷ 3,009개 = intent당 평균 32개
- 저빈도 intent는 7~10개밖에 없다
- 공공 데이터셋 + LLM 증강으로 최소 10,000개 이상 필요

## 해결 계획

### Step 1: 학습 데이터 대폭 확장 (3,009 → 10,000+)
- **공공 데이터:** AIHub 스마트홈 데이터, 한국어 대화 데이터 검색
- **LLM 증강:** Claude/GPT로 94개 intent별 50~100개 발화 생성 (오프라인, 1회성)
- **STT 오류 + 간접 표현:** 자동 변형 생성

### Step 2: 테스트셋 분리
- 학습 데이터와 **완전히 겹치지 않는** 테스트셋 1,000개 이상 새로 생성
- 간접 표현, STT 오류, 정상 발화 골고루 포함

### Step 3: 재학습 + 재평가
- 확장 데이터로 학습
- 새 테스트셋으로 평가 → 이게 진짜 정확도
