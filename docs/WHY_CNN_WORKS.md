# 왜 4.4M CNN이 간접 표현을 잡고, KoELECTRA는 못 했나

## 결론부터

**모델이 아니라 데이터가 문제였다.**

KoELECTRA가 간접 표현을 못 한 건 모델 능력 부족이 아니라, 학습 데이터에 간접 표현이 없었기 때문이다. 우리 4.4M CNN도 간접 표현 데이터 없이 학습하면 똑같이 실패한다.

---

## 1. KoELECTRA가 실패한 이유

### 다른 팀의 접근

```
학습 데이터:
  "에어컨 켜줘" → ac_on
  "에어컨 켜주세요" → ac_on
  "거실 에어컨 켜줘" → ac_on
  ... (직접 명령만)

테스트:
  "찜통이야" → ???  (학습 데이터에 없는 패턴)
  "코가 시려" → ???  (학습 데이터에 없는 패턴)
```

KoELECTRA(14M params, 6-layer Transformer)는 충분히 강력한 모델이다.
하지만 **학습 데이터에 "찜통이야"→ac_on 같은 간접 표현이 없으면**, 모델이 아무리 크고 좋아도 이걸 추론할 수 없다.

### "KoELECTRA는 문맥을 이해하니까 간접 표현도 되지 않나?"

이건 흔한 오해다.

KoELECTRA의 사전학습은 **한국어 텍스트의 일반적인 문맥 관계**를 학습한 것이다.
"찜통"이 "더운 환경"이라는 것은 알 수 있다.
하지만 **"더운 환경 → 에어컨을 켜야 한다 → ac_on intent"** 이 연결고리는 사전학습에 없다.

이건 **상식 추론(commonsense reasoning)**이지, 텍스트 이해가 아니다.
LLM(GPT-4, Claude 등)이 이걸 할 수 있는 건 수조 개의 대화 데이터에서 "덥다 → 에어컨" 패턴을 이미 봤기 때문이다.

---

## 2. 우리 CNN이 성공한 이유

### 학습 데이터에 답이 있다

```
학습 데이터:
  "에어컨 켜줘" → ac_on        (직접 명령)
  "찜통이야" → ac_on            (간접 표현 ★)
  "더워 죽겠어" → ac_on          (간접 표현 ★)
  "땀이 나" → ac_on             (간접 표현 ★)
  "후덥지근해" → ac_on           (간접 표현 ★)

테스트:
  "찜통이야" → ac_on  ✓  (학습 데이터에 있는 패턴)
```

**모델이 "찜통"의 의미를 이해하는 게 아니다.**
"찜통이야"라는 토큰 시퀀스가 ac_on 레이블과 함께 학습되었기 때문에, 같은 입력이 들어오면 ac_on을 출력하는 것이다.

### 그러면 학습 데이터에 없는 간접 표현은?

맞다. **학습 데이터에 없는 간접 표현은 못 맞춘다.**

예를 들어 "에어컨 켜줘" 계열만 학습시키고 "찜통이야"를 안 넣으면, "찜통이야"를 입력했을 때 엉뚱한 intent가 나온다.

하지만 이건 LLM도 아닌 이상 모든 NLU 모델이 동일하다. KoELECTRA든 BERT든 CNN이든.

### 핵심: "학습 데이터에 넣으면 되는 건 KoELECTRA도 마찬가지 아닌가?"

**맞다.** KoELECTRA에 같은 간접 표현 데이터를 넣고 학습시켜도 됐을 것이다.

다른 팀이 실패한 건 **KoELECTRA의 한계가 아니라, 간접 표현 데이터를 안 만든 것**이 원인이다.

---

## 3. 그러면 왜 거대한 KoELECTRA 대신 작은 CNN을 쓰나?

### 크기와 속도

| 모델 | Params | ONNX 크기 | NPU NB | 추론 시간 |
|------|--------|----------|--------|----------|
| KoELECTRA-Small | 14M | ~36MB | NPU 실패 | CPU 80ms |
| **PureCNN 4L** | **4.4M** | **~1.2MB** | **~300KB** | **NPU ~1ms** |

### NPU 호환성

T527 NPU에서 Transformer(Self-Attention)는 양자화 에러 누적으로 동작하지 않는다.
9가지 양자화 방식을 시도했지만 전부 실패했다 (NPU_NLU_EXPERIMENTS.md 참조).

CNN은 Conv1d + ReLU + BatchNorm만 사용하므로 NPU에서 완벽하게 동작한다.

### 정확도는?

| 모델 | 테스트 정확도 | 간접 표현 | NPU |
|------|------------|---------|-----|
| KoELECTRA (간접표현 데이터 없이) | 높음 (직접 명령) | **실패** | **실패** |
| PureCNN 4L (간접표현 데이터 포함) | **100% (682/682)** | **100% (31/31)** | **동작 (~1ms)** |

간접 표현 정확도는 모델 크기가 아니라 학습 데이터가 결정한다.
같은 데이터를 넣으면 4.4M CNN이나 14M KoELECTRA나 결과가 비슷하다.

## 4. CNN이 간접 표현을 학습하는 원리

### 토크나이저가 핵심

BERT 토크나이저(WordPiece)가 한국어를 서브워드로 분리한다:

```
"찜통이야"  → ["찜", "##통", "##이", "##야"]
"더워 죽겠다" → ["더", "##워", "죽", "##겠", "##다"]
"에어컨 켜줘" → ["에어컨", "켜", "##줘"]
```

CNN은 이 토큰 시퀀스에서 n-gram 패턴을 학습한다:

- kernel=3: "찜-통-이" 패턴 → ac_on과 연관
- kernel=5: "더-워-죽-겠-다" 패턴 → ac_on과 연관
- kernel=7: 더 넓은 문맥 패턴

### 왜 4 Layer면 충분한가

Intent classification은 본질적으로 **텍스트 분류(text classification)** 문제다.
문장 전체의 의미를 하나의 레이블로 매핑하는 것이다.

이 문제의 특성:
1. **입력이 짧다** — 스마트홈 발화는 평균 5~10 토큰
2. **클래스가 적다** — 50개 intent
3. **패턴이 명확하다** — "켜줘"→on, "꺼줘"→off, "올려줘"→up

이런 조건에서는 Self-Attention(장거리 의존성)이 필요 없다.
짧은 문장의 로컬 패턴만 잡으면 되므로 CNN의 convolution이 더 효율적이다.

실험으로도 확인했다:

```
CNN 4L:  4.40M params → val_acc 0.9852
CNN 8L:  6.51M params → val_acc 0.9847  (오히려 하락)
CNN 24L: 35.03M params → val_acc 0.9837  (더 하락)
```

모델을 키우면 오히려 과적합되어 성능이 떨어진다.

## 5. 한계

### 이 접근이 안 되는 경우

1. **학습 데이터에 없는 완전히 새로운 간접 표현**
   - "에어컨 좀 돌려" → 학습에 없으면 실패 가능
   - 해결: 추가 데이터로 재학습 (빠름, 30초)

2. **복합 의도 (multi-intent)**
   - "불 끄고 에어컨 켜줘" → 하나의 intent만 반환
   - 해결: 문장 분리 후 각각 NLU 처리, 또는 multi-label 분류로 확장

3. **문맥 의존 (이전 대화 참조)**
   - "거기도 켜줘" → "거기"가 어디인지 이전 대화를 봐야 함
   - 해결: 대화 이력 관리 (NLU 범위 밖)

4. **자유형 질문**
   - "에어컨이랑 공기청정기 차이가 뭐야?" → 정보 검색 질문
   - 해결: LLM 필요 (하이브리드 구조)

### LLM이 정말 필요한 경우

- 학습 데이터로 커버할 수 없을 만큼 **발화 패턴이 무한**할 때
- **자유형 대화**가 필요할 때 ("오늘 뭐 해먹지?")
- **문맥 추론**이 필요할 때 ("그거 말고 다른 거")

스마트홈 제어는 발화 패턴이 제한적이므로, 데이터만 잘 만들면 LLM 없이 가능하다.

## 6. 요약

```
Q: 왜 KoELECTRA는 간접 표현이 안 됐나?
A: 학습 데이터에 간접 표현이 없었다. 모델 문제가 아니라 데이터 문제.

Q: 왜 4.4M CNN이 간접 표현을 맞추나?
A: 학습 데이터에 "찜통이야"→ac_on 같은 쌍을 넣었기 때문.

Q: CNN이 "찜통"의 의미를 이해하는 건가?
A: 아니다. 토큰 패턴 매칭이다. 이해가 아니라 암기에 가깝다.

Q: 그러면 KoELECTRA에 같은 데이터 넣어도 되지 않나?
A: 된다. 하지만 KoELECTRA는 T527 NPU에서 안 돌아간다.
   CNN은 300KB NB로 1ms만에 돌아간다.

Q: LLM이 필요한 경우는?
A: 발화 패턴이 무한하거나, 자유형 대화가 필요할 때.
   스마트홈 제어(50개 intent)는 LLM 없이 가능.
```

| 직접 명령 | O | O | O |
| 간접 표현 (데이터 있을 때) | O | **O** | O |
| 간접 표현 (데이터 없을 때) | X | X | **O** |
| T527 NPU | **X** | **O (~1ms)** | X |
