우리 Test Suite 3,043개에서는 100%이지만, **처음 보는 데이터에서는 75.5%**.

## 한계 1: 모델이 확신하면서 틀림 (High-Confidence Error)

오답 376개 중 **68% (256건)가 confidence 0.9 이상**. 모델이 "확실히 이거다"라고 판단하면서 틀림.

→ confidence threshold로 잡을 수 없음. 모델이 자기가 모르는 걸 모름.

원인: 학습 데이터에 없는 표현이 들어오면, ko-sbert 임베딩이 가장 가까운 학습 데이터로 매핑 → 높은 확신으로 오답.

## 한계 2: 우리 체계에 없는 디바이스 (24건)

KoELECTRA에는 있지만 우리 fn 20개에 없는 것:
- **콘센트/플러그/멀티탭** (24건) → light_control로 매핑됐지만 실제로는 별도 디바이스
- **루버** (51회) → 커튼이랑 비슷하지만 다른 기기
- **가스밸브 차단** vs **가스 잠금** — 같은 건지 다른 건지 경계 모호

→ fn 20개 체계가 실제 월패드 기능을 다 커버하지 못함

## 한계 3: known→unknown 누출 (137건, 36%)

가장 큰 문제. 정상 발화가 unknown으로 빠짐:
- "통화 내역 확인해주세요" → unknown (home_info여야 함)
- "카드 삭제해" → unknown (security_mode여야 함)
- "멀티탭 켜주세요" → unknown (light_control여야 함)
- "비밀번호 확인하고 싶어" → unknown (system_meta여야 함)
- "차량 출입 기록 보여줘" → unknown (vehicle_manage여야 함)

원인: 
- **"통화", "카드", "멀티탭", "비밀번호 확인"** 같은 키워드가 학습 데이터에 없음
- unknown 학습 데이터에 비슷한 패턴("예약해줘", "주문해줘" 등)이 있어서 "~해줘" 패턴을 unknown으로 연관
- 모델이 **키워드 기반으로 학습**하다 보니, 키워드 하나가 바뀌면 완전히 다른 결과

## 한계 4: 완전 오분류 (138건, 37%)

아예 관련 없는 fn으로 분류:
- "관리실 호출해줘" → elevator_call ("호출" 키워드가 엘리베이터와 연관)
- "재택 모드 설정해줘" → energy_query ("설정" 키워드 혼동)
- "단지 소식 있나요" → news_query ("소식" = 뉴스 연상)
- "냉방기 온도 올려줘" → heat_control ("온도 올려" 키워드가 난방과 연관)

원인: 모델이 **문맥이 아닌 키워드에 과의존**. "호출"=엘리베이터, "온도 올려"=난방으로 고정.

우리 증강 데이터는:

- "잠깐", "얼른" 같은 자연스러운 구어 부사가 거의 없음
- "해줄래요", "줄래요" 같은 존댓말 어미 다양성 부족
- **규칙 기반 증강의 근본적 한계** — 같은 패턴의 반복이지 새로운 표현이 아님

## 한계 6: fn 경계 혼동 (84건, 22%)

같은 그룹 내 fn 구분 실패:
- ac_control ↔ heat_control: "냉방기 온도 올려줘" → heat ("온도 올려"가 heat에 편향)
- vent_control ↔ weather_query: "공기 좀 바꿔줘" → weather ("공기"가 미세먼지와 연관)
- home_info ↔ system_meta: "화면 밝기 조절" → home_info vs system_meta 경계 모호

원인: 유사한 fn 간 의미 경계를 **데이터 양으로만 학습** → 데이터 편향 방향으로 치우침

## 근본 원인 정리

1. **규칙 기반 증강의 한계**: 어미 변형, room 교차, STT 오류 주입만으로는 **실제 사용자의 발화 다양성을 커버 못함**. "잠깐 플러그 전원 빼줘" 같은 표현은 규칙으로 생성 불가.

2. **키워드 과의존**: ko-sbert 임베딩이 의미를 잡아주지만, CNN이 결국 **n-gram 패턴 매칭**에 의존. "호출"=엘리베이터, "온도 올려"=난방 같은 단순 연관이 형성됨.

3. **fn 체계 불완전**: 콘센트, 루버, 통화, 카드 관리 등 실제 월패드에 있는 기능이 fn 20개에 포함 안 됨.

4. **unknown 과학습**: "피자 주문", "노래 틀어" 같은 OOD 패턴을 과하게 학습 → "~해줘/~해주세요" 패턴이 조금만 낯설면 unknown으로 빠짐.

5. **학습 데이터 = 자기 증강 데이터**: 자기가 만든 데이터로 자기를 테스트하니 100%. 외부 데이터가 들어오면 실체 노출.

## 해결 방향

1. **KoELECTRA 데이터의 exec/dir 라벨을 수작업 보정 후 병합** — 이번 regression은 라벨 불일치가 주원인. fn은 대체로 맞음.
2. **fn 체계 확장** — 콘센트/루버/통화/카드 등 누락 기능 추가 검토.
3. **LLM 기반 증강** — 규칙 증강 대신 LLM으로 자연스러운 변형 생성 (잠깐/얼른/해줄래요 등).
4. **unknown threshold 조정** — known→unknown 누출이 심하므로 unknown 학습 비율 줄이거나, 경계를 느슨하게.
5. **cross-validation 루틴 도입** — 자체 test suite만이 아닌, 독립 외부 데이터(KoELECTRA val)로 항상 교차 검증.
